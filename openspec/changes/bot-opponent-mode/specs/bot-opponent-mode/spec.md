## ADDED Requirements

### Requirement: Bot opponent match setup
The system SHALL support a local solo match mode in which one human player competes against one AI-controlled opponent. The AI opponent SHALL occupy the non-human player slot for the match.

#### Scenario: Start bot match
- **WHEN** the player starts a solo bot match
- **THEN** the match initializes with one human-controlled player and one AI-controlled opponent

#### Scenario: Opponent is AI controlled
- **WHEN** the match begins
- **THEN** the opponent's actions are generated by the bot rather than direct player input

### Requirement: Bot action generation
The system SHALL generate opponent actions for each tick based on the current game state and available resources. The bot SHALL only queue actions that pass normal game validation. The bot SHALL evaluate up to 3 actions per tick when resources allow.

#### Scenario: Bot queues legal actions
- **WHEN** the bot has resources and the current state allows a valid action
- **THEN** the bot queues at least one legal action before the tick resolves

#### Scenario: Bot skips invalid actions
- **WHEN** the bot would select an action that fails validation
- **THEN** the action is not applied and the tick continues with the remaining valid actions

### Requirement: Bot expansion strategy
The bot SHALL expand toward the enemy spawn by generating candidate positions from frontier nodes (friendly nodes closest to the enemy). Candidate positions SHALL be computed by projecting from frontier nodes toward the enemy spawn point within the map's maximum connection distance.

#### Scenario: Bot expands toward enemy
- **WHEN** the bot has connected friendly nodes
- **THEN** candidate placement positions are biased toward the enemy spawn point rather than placed randomly across the map

#### Scenario: Bot uses frontier nodes
- **WHEN** the bot has multiple friendly connected nodes
- **THEN** only the nodes closest to the enemy spawn are used as anchor points for expansion

### Requirement: Bot strategic node selection
The bot SHALL prioritize node types based on game phase and strategic context:
- Generators SHALL be prioritized early and when the bot has few generators, to accelerate income.
- Turrets SHALL be prioritized when enemy nodes are within attack range.
- Shields SHALL be prioritized near friendly turrets and in contested territory.
- The bot SHALL deprioritize purchases that would leave it unable to afford a generator.

#### Scenario: Bot builds income first
- **WHEN** the bot has fewer than 3 generators and can afford one
- **THEN** generator placement receives a higher score than other node types

#### Scenario: Bot places turrets near enemy nodes
- **WHEN** enemy connected nodes exist within turret attack range of a candidate position
- **THEN** turret placement at that position receives a significant score bonus

### Requirement: Bot resource-aware spending
The bot SHALL track cumulative resource cost across multiple actions in a single tick. The bot SHALL NOT select actions whose combined cost exceeds the bot's available resources.

#### Scenario: Bot respects budget across multiple actions
- **WHEN** the bot selects 3 actions in one tick
- **THEN** the total cost of all selected actions does not exceed the bot's current resources

### Requirement: Bot decisions are deterministic
Given the same game state and configuration, the bot opponent SHALL produce the same actions. The bot SHALL NOT use random number generation or non-deterministic ordering.

#### Scenario: Identical state yields same move
- **WHEN** two matches reach the same game state with the same configuration
- **THEN** the bot chooses the same actions in both matches

#### Scenario: Stable tie-breaking
- **WHEN** two candidate actions have the same score
- **THEN** ties are broken by node type (lexicographic), then position x-coordinate, then y-coordinate
